# 할루시네이션 게이트 계획 — "LLM은 지각, 규칙이 제어흐름" (2026-07-12)

> 배경: 지금까지 할루시네이션을 **모델 교체/확대**(Ollama 10종 비교 → DGX 32B → 파인튜닝)로
> 풀려고 했다. SCPC 2026 대회에서 실증한 교훈을 적용해 방향을 뒤집는다:
> **모델은 그대로 두고, LLM이 결정하는 영역을 규칙으로 줄인다.**
>
> 근거가 된 대회 실측: ①"만들지 마/자세히 읽어" 류 지시문은 성능을 못 움직인다
> (0.645→0.645, 지시≠메커니즘) ②LLM은 지각(무엇이 적혀 있나)은 정확하고 제어흐름
> (무엇이 무엇을 이기나)에서 샌다 ③결정론 게이트+평가 하네스 규율이 0.705→0.8636을 만들었다.

## 0. 현 파이프라인 진단 (backend/app/ai/)

```
query → retrieve_room_chunks (키워드 겹침 ≥1, confidence=0.75 하드코드)
      → build_prompt_package (지시문 + persona + evidence)
      → provider.generate (Ollama)
      → 응답 무검증 통과 ← ★ 모든 할루가 여기로 나간다
```

할루 유입점 3개:

| # | 유입점 | 현재 방어 | 문제 |
|---|---|---|---|
| 1 | **검색 품질** — 겹침 1단어면 통과, 무관 chunk 혼입, 빈 결과여도 LLM 호출 | 없음 | 약한 근거 + 질문 → 모델이 지어냄 |
| 2 | **프롬프트 지시문만으로 방어** ("do not invent…") | prompt_builder의 지시문 | 대회에서 실증된 무효 수단. 지시는 메커니즘이 아니다 |
| 3 | **출력 무검증** — `</think>` 누수, `# Identity` 블록 누수, 영어 전환, 근거 오독, saved_at↔사건날짜 혼동 (전부 5-11~5-19 실측에서 관찰됨) | 없음 | 관찰된 실패가 전부 **규칙으로 100% 잡히는 종류**인데 안 잡고 있다 |

핵심 관찰: 이미 갖고 있는 자산이 크다 — **5질문 평가셋**(거절형·모순형 포함), **10모델 실측
JSON**, **fallback 템플릿**(`render_demo_answer`). 이걸 게이트와 채점기의 종잣돈으로 쓴다.

## 1. 목표 아키텍처

```
query ─→ [입력 게이트: answerability 래더]           ← 규칙 (Phase 2)
           ├─ REFUSE   → 거절 템플릿 (LLM 호출 없음 = 할루 0)
           ├─ CLARIFY  → 되묻기 템플릿 (LLM 호출 없음)
           ├─ QUOTE    → 기록 인용 템플릿 (LLM 호출 없음)
           └─ ANSWER   → fact-sheet 추출(규칙) → LLM은 문체 렌더링만  (Phase 3)
                              ↓
         [출력 게이트: 누수필터 → 근거대조 → 날짜검사]  ← 규칙 (Phase 1)
           ├─ 통과     → 사용자에게
           └─ 실패     → QUOTE 템플릿으로 강등 (환각 문장 절대 통과 금지)
```

SCPC의 control 래더(proceed/amend/hold/ask)와 정확히 동형이다:
ANSWER=proceed, QUOTE=amend(범위 축소), REFUSE=hold, CLARIFY=ask.

## Phase 0 — 평가 하네스 먼저 (반나절) ★모든 것의 전제

대회 규율 그대로: **게이트 없이 개선 없다.** 고치기 전에 재는 도구부터.

- `research_notes/`의 5질문을 **골든셋 ~30문항**으로 확장. 각 문항에 라벨:
  - `answerable`(기록에 답 있음) / `unanswerable`(없음→거절이 정답) / `conflict`(모순 제시가 정답)
  - 정답 원자(atom): 기록에 실재하는 날짜·이름·장소·수치 목록
- **규칙 기반 자동 채점기** `backend/tools/eval_hallucination.py`:
  - **미지원-원자율**: 응답에서 정규식으로 날짜/수사/고유명사 후보를 뽑아, 검색된 evidence
    텍스트에 없는 원자의 비율 (핵심 지표)
  - **거절 정확도**: unanswerable 문항에서 "기록에 없다"류로 답했는가 (cue 리스트 판정)
  - **누수율**: `</think>`, `# Identity`, `Retrieved Evidence`, `tags=`, `confidence=` 등
    패턴 검출
  - **언어 위반율**: 한글 비율 임계
- 산출물: 모델별 **baseline 표** (게이트 없음 상태). 이후 모든 Phase는 이 표의 전/후로 판정.

수용 기준: 채점기가 기존 실측 JSON(5-19)을 재채점해서 그때의 수동 관찰(누수·거절실패)과
일치하면 합격.

## Phase 1 — 출력 게이트 (가장 싼 최대 효과, 1일)

`backend/app/ai/gates/output_gate.py` — 순수 함수, stdlib만, 결정론.

1. **누수 필터**: `</think>`·reasoning 블록·프롬프트 라벨·tags/confidence 문자열 제거,
   제거 후 잔여물이 비면 게이트 실패 처리
2. **근거 대조 (핵심)**: 응답을 문장 단위로 쪼개고, 문장 속 검증가능 원자
   (날짜 `\d{1,4}[년./-]…`, 수사, 한글 고유명사 후보=조사 제거 후 2~5음절 명사구가
   evidence에 등장하는지)를 대조.
   - 원자가 evidence에 없음 → 해당 **문장 드랍** 또는 "기록에는 확인되지 않습니다"로 치환
   - 드랍 후 남는 문장이 없으면 → QUOTE 템플릿으로 강등
3. **날짜 의미 검사**: 응답이 사건 날짜를 단정하는데 evidence엔 saved_at뿐이면
   "기록 저장일 기준" 문구를 강제 삽입 (5-11 실측의 고질 혼동)
4. **모순 검사**: 같은 원자 유형(날짜 등)에 대해 evidence 두 개가 다른 값을 가지면
   양립 제시 템플릿 강제

`demo_service.build_demo_chat_response`의 `answer = …` 한 줄 뒤에 끼우면 되는 위치다.
기대 효과: 관찰된 실패 모드(누수·근거 오독·무근거 세부) 대부분이 여기서 죽는다.
**모델 불문이라 gemma4:e2b 그대로 유지 가능** — 이게 이 방향의 요점.

## Phase 2 — 입력 게이트: answerability 래더 (1~2일)

`backend/app/ai/gates/answer_router.py`

1. **검색 점수 실질화**: 가짜 `confidence=0.75` 폐지 → `겹침 단어 수 / 질의 단어 수`
   (coverage)와 매칭 단어 목록을 chunk에 실어 보냄
2. **질의 유형 cue 리스트** (SCPC `_CUES` 방식, 열거+dev검증):
   - 사실질문형(언제/어디/누가/얼마), 요약형, 조언·회고형(기록 불요), 잡담형
3. **래더** (위에서부터, 첫 매치 승):
   - 검색 0건 ∧ 사실질문형 → **REFUSE** (기존 `render_demo_answer` 확장판 사용)
   - 검색 있음 ∧ coverage < τ ∧ 사실질문형 → **CLARIFY** ("어느 기록을 말하는지…")
   - 질문의 핵심 원자(이름·날짜)가 어떤 chunk에도 없음 → **REFUSE-부분**
     ("관련 기록은 있으나 물으신 ○○는 확인되지 않습니다" + chunk 인용)
   - 요약형 ∧ 검색 있음 → **ANSWER**
   - 조언·회고형 → **ANSWER** (근거표시 완화, 단 사실 원자는 Phase 1 대조 유지)
4. REFUSE/CLARIFY/QUOTE는 **LLM을 아예 호출하지 않는다** — 호출 안 한 답변의 할루율은
   정의상 0. 골든셋의 unanswerable 문항이 이 경로 커버리지를 측정한다.

## Phase 3 — LLM 역할 축소: fact-sheet 렌더링 계약 (2~3일, 선택적 고도화)

ANSWER 경로에서도 LLM에게 자유 작문을 주지 않는다:

1. 규칙이 chunk에서 **fact-sheet** 추출: `[{atom, 값, 출처 chunk#}]` (날짜·이름·사건 구절)
2. LLM 입력 = fact-sheet + persona + 질문. 계약: "이 사실 목록만으로, 문장마다 근거 번호를
   `[1]`처럼 붙여 답하라"
3. 출력 게이트가 문장별 인용 번호를 검증 — 인용 없는 문장/틀린 인용 문장은 드랍
4. 소형 모델이 계약을 자주 어기면(실측으로 판단) 문장별 사후 대조(Phase 1 방식)로 폴백

이러면 LLM의 남은 역할 = **문체(persona)와 요약 구성뿐**. 사실의 출처는 전부 규칙 경로.

## Phase 4 — 회귀 게이트 고정 (SCPC check.py 이식, 반나절)

`backend/tools/check_gates.py`:
- 골든셋 지표 ratchet: 미지원-원자율 상한, 거절 정확도 하한, 누수율 0
- **metamorphic**: ①evidence 전부 제거 → 반드시 REFUSE ②모순 chunk 주입 → 양립 제시
  ③evidence 순서 셔플 → 답 불변 ④`</think>` 포함 가짜 provider 응답 주입 → 누수 0
- 결정성: 같은 입력 2회 → 동일 출력 (게이트 층은 완전 결정론이어야)

## 실행 순서와 예상 효과

| 순서 | 작업 | 공수 | 기대 효과 |
|---|---|---|---|
| 0 | 골든셋+채점기, baseline 측정 | 반나절 | 이후 모든 판단의 눈 |
| 1 | 출력 게이트 | 1일 | 누수 100% 제거, 무근거 원자 문장 차단 |
| 2 | 입력 래더 | 1~2일 | unanswerable에서 할루 0 (LLM 미호출) |
| 4 | 회귀 게이트 | 반나절 | 재발 방지 |
| 3 | fact-sheet 계약 | 2~3일 | ANSWER 경로 잔여 할루 축소 (측정 후 결정) |

Phase 3은 0~2의 실측 결과를 보고 착수 여부를 정한다 (0~2만으로 골든셋 할루율이
충분히 떨어지면 과설계 회피).

## 대회에서 가져오는 규율 3개

1. **게이트 먼저, 수정은 나중** — 재지 않은 개선은 개선이 아니다
2. **cue 리스트는 dev(골든셋) 오탐 0 검증 후에만 추가** — 열거는 부끄러운 게 아니다,
   검증 없는 열거가 문제다
3. **모든 게이트 층은 순수 함수·stdlib·결정론** — 테스트 가능하고, 모델을 바꿔도 살아남는다
