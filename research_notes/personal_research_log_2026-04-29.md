# 개인연구록 초안

## 연구 주제

온디바이스 LLM 선택 구조와 Markdown 기반 페르소나 유지 방식을 고려한 디지털 유산 서비스의 하이브리드 분산 추론 아키텍처 재설계

## 작성용 본문 초안

이번 개인 연구에서는 기존의 로컬 서버 중심 디지털 유산 서비스 구조를 재검토하고, 온디바이스 모델을 활용한 하이브리드 분산 추론 구조의 적용 가능성을 중심으로 설계를 재정리하였다. 기존 프로젝트는 모바일 앱과 FastAPI 백엔드를 기반으로 업로드 메타데이터, 가족방, 클라우드 저장 구조를 우선 구현하는 방향으로 진행되어 왔으나, 최근 공개되는 경량 온디바이스 모델들의 발전 속도와 디지털 유산 서비스의 장기 보존성 요구를 함께 고려할 필요가 있다고 판단하였다.

우선, 온디바이스 모델의 가능성을 검토하기 위해 Google AI Edge 계열 공개 자료와 Google AI Edge Gallery 구조를 조사하였다. 조사 결과, Google의 LiteRT-LM은 온디바이스 LLM 실행을 위한 공개 프레임워크로 제시되고 있으며, 최근에는 Gemma 4 계열 모델 지원을 전면에 내세우고 있었다. 또한 공개된 Android용 Gallery allowlist 기준으로 Gemma-4-E2B-it와 Gemma-4-E4B-it 모델이 실제 선택 가능한 온디바이스 후보로 포함되어 있었고, 이미지 및 오디오 입력과 최대 32K 컨텍스트 길이를 지원하는 것으로 정리되어 있었다. 반면 iOS 쪽 공개 allowlist는 Gemma 3n 중심으로 확인되어, 현재 단계에서는 Gemma 4 온디바이스 실험이 Android 환경에서 더 현실적이라고 판단하였다.

이러한 조사 결과를 바탕으로, 본 프로젝트의 추론 구조를 단일 모델 중심으로 고정하기보다 모델 교체 가능성을 전제로 한 구조로 바꾸는 것이 더 적절하다고 판단하였다. 최근 공개되는 모델의 수명 주기가 매우 짧고, 특정 모델을 중심으로 서비스를 설계하면 추후 모델 교체 시 프롬프트 구조, 페르소나 반영 방식, 검색 파이프라인을 다시 손봐야 할 가능성이 크기 때문이다. 따라서 메인 추론 모델과 사용자별 선택 모델을 분리하는 방향을 연구 대상으로 삼았다.

구체적으로는 가족 단위의 정본 데이터와 권한 체계는 공통 레이어에서 유지하고, 각 개인은 자신의 기기에서 별도의 온디바이스 모델과 페르소나를 선택해 사용할 수 있는 하이브리드 분산형 Sovereign AI 구조를 검토하였다. 이 구조에서 가족 금고 노드 또는 로컬 서버는 원본 유산 데이터, OCR/STT 기반 메타데이터, 공용 인덱스, 권한 필터링, 백업 및 복구 기준점을 담당한다. 반면 개인 기기는 최근 열람 기록과 일부 동기화된 인덱스를 기반으로 빠른 RAG 응답을 수행하며, 필요에 따라 온디바이스 Gemma 모델을 통해 오프라인 또는 저지연 질의를 처리할 수 있다. 이를 통해 "한 기기가 노후되거나 교체되면 유산 데이터와 대화 경험이 모두 무너지는 문제"를 줄이면서도, 데이터 주권과 개인화된 경험을 동시에 확보하는 방향을 건의할 수 있다고 보았다.

또한 페르소나 유지 방식 역시 단일 프롬프트 문자열이 아니라 Markdown 자산 기반으로 관리하는 방안을 검토하였다. 이에 따라 인물 단위의 정보를 `identity.md`, `voice.md`, `timeline.md`, `memory_policy.md` 등으로 나누어 관리하는 구조를 건의하였다. 이 방식은 모델이 교체되더라도 페르소나 자산을 재사용할 수 있고, 가족 구성원이 내용을 직접 검수하거나 수정하기에도 유리하다. 특히 디지털 유산 서비스에서는 고인의 성격, 화법, 기억의 경계, 추정 금지 규칙을 지속적으로 조정할 필요가 있기 때문에, 페르소나를 문서 자산으로 남기는 방식을 장기적 관리 방안으로 제안할 수 있다고 판단하였다.

검색 및 질의응답 구조에 대해서는 RAG를 단일 중앙 검색으로만 보지 않고, 다층 retrieval 구조로 재정리하는 방향을 검토하였다. 1차로는 개인 기기 내부의 최근 기록 캐시와 즐겨찾기 기반 retrieval을 수행하고, 2차로는 가족 금고 노드의 공용 정본 인덱스에서 권한 필터를 거친 기록을 불러오며, 필요 시 장기 백업 스냅샷에서 복원하는 구조를 구상하였다. 이와 같은 구조는 단순 속도 개선을 넘어, 기기 교체 시 데이터와 페르소나를 재동기화할 수 있는 기반이 될 수 있다는 점에서 후속 구현 방향으로 건의할 수 있다.

실제 구현 가능성 검토를 위해 원본 Ambient Legacy 저장소 구조를 다시 읽고, 완전한 모델 연동 이전에 넣을 수 있는 최소한의 추상화 레이어를 설계하였다. 구체적으로는 백엔드 내부에 `model_registry`, `persona_loader`, `prompt_builder` 역할의 AI orchestration 계층을 추가하는 방식을 구상하였다. 이 레이어는 특정 모델에 종속되지 않고, 사용자가 선택한 모델 프로필, Markdown 페르소나 묶음, retrieval evidence를 조합하여 최종 추론 입력 패키지를 구성하는 역할을 담당한다. 이를 통해 EXAONE, Gemma 4, Gemma 3n 등 서로 다른 모델 계열을 나중에 붙이더라도 상위 서비스 구조를 크게 흔들지 않도록 설계할 것을 건의하였다.

추가로, 실제 구현 가능성을 더 구체적으로 확인하기 위해 provider adapter 구조를 별도로 설계하였다. 이에 따라 Gemma 계열과 EXAONE 계열을 각각 독립적인 provider로 두고, 상위 서비스는 `model_id`에 따라 적절한 adapter를 선택하도록 구성하였다. 또한 앱 쪽에서는 모델 선택 UI와 페르소나 선택 UI를 먼저 구성하고, 백엔드의 `/ai/models`, `/ai/personas`, `/ai/chat-demo` API를 통해 실제 선택값과 질의가 연결되도록 실험 구조를 정리하였다. 이 과정에서 온디바이스 모델과 서버 모델을 동일한 상위 인터페이스로 묶을 수 있다는 점을 확인하였으며, 향후 실제 Gemma 런타임 또는 EXAONE 로컬 서버를 붙일 때도 상위 서비스 구조를 크게 바꾸지 않고 확장할 수 있다고 보았다.

실제 구현 범위는 목요일 시연을 위한 프로토타입 수준으로 제한하였다. 앱에서는 Chat 탭에 모델 선택 UI, 페르소나 선택 UI, 데모 데이터 준비 버튼, 추천 질문, provider 상태 및 retrieval evidence 표시 영역을 추가하였다. 백엔드에서는 `/api/v1/ai/models`, `/api/v1/ai/personas`, `/api/v1/ai/runtime-status`, `/api/v1/ai/demo-bootstrap`, `/api/v1/ai/chat-demo` API를 추가하였다. 또한 Gemma, EXAONE, mock, Ollama provider adapter를 구성하고, persona markdown 로더와 prompt builder를 통해 `선택 모델 + 페르소나 문서 + 가족방 기록 근거`를 하나의 추론 입력 패키지로 묶는 구조를 구현하였다. Mac 환경에서는 Ollama 기반 `gemma4:e2b` 모델을 다운로드하여 로컬 추론 응답을 확인하였고, 앱은 Mac backend 주소를 바라보도록 설정하여 데모 계정 기반 흐름을 실험할 수 있도록 구성하였다. 단, 갤럭시 앱 내부에서 모델 파일을 직접 다운로드하고 온디바이스로 실행하는 기능은 이번 구현 범위에 포함하지 않았다.

이와 함께 목요일 데모 시연을 염두에 두고, 구현 범위를 “완전한 AI 성능 검증”보다 “구조적 시연 흐름 확보”에 맞추어 조정하였다. 구체적으로는 앱의 Chat 탭에서 모델과 페르소나를 각각 선택한 뒤, 백엔드에 질문을 전송하면 가족방의 샘플 기록을 바탕으로 retrieval evidence와 응답을 함께 반환하는 데모 흐름을 구성하였다. 또한 실제 발표 중 데이터가 비어 보이는 리스크를 줄이기 위해, `demo-bootstrap` API를 별도로 설계하여 한 번의 요청으로 데모 가족방과 샘플 텍스트, 이미지, 영상 기록을 자동 생성하는 구조를 추가하였다. 이를 통해 사용자는 로그인 후 별도 수작업 없이 바로 AI 데모와 저장소 데모를 연결해 볼 수 있는 상태를 만드는 것을 목표로 하였다.

추론 품질 측면에서는 아직 실제 Gemma 또는 EXAONE 런타임과의 완전한 연결보다 provider 계약과 상위 호출 구조를 먼저 확정하는 것이 더 중요하다고 판단하였다. 이에 따라 mock inference server와 provider contract 문서를 함께 정리하여, 현재는 endpoint가 없더라도 구조 검증은 가능하고, 이후 실제 모델 서버가 준비되면 endpoint URL만 교체하여 점진적으로 실연결할 수 있도록 하였다. 이러한 방식은 디지털 유산 서비스에서 모델의 교체 속도가 빠르다는 문제를 완화하고, 서비스 자산을 특정 모델 구현에 고정하지 않는다는 점에서 의미가 있다고 보았다.

다만 추가 검토 과정에서, 온디바이스 LLM을 모든 사용자에게 기본값으로 두는 것은 현실적인 사용자 경험 측면에서 위험할 수 있다는 점을 확인하였다. 예를 들어 고령 사용자가 보급형 갤럭시 A 시리즈와 같은 기기를 사용하는 경우, 2B급 경량 모델이라 하더라도 양자화된 모델 파일, 런타임 메모리, 긴 RAG 컨텍스트, 발열, 배터리 소모, OS의 백그라운드 앱 정리 정책이 함께 작용할 수 있다. 이 경우 단순히 "모델이 실행된다"는 것과 "사용자가 다른 앱을 함께 사용하면서도 버벅임 없이 자연스럽게 대화할 수 있다"는 것은 다른 문제라고 판단하였다. 특히 디지털 유산 서비스의 주요 사용자 중에는 고사양 기기를 쓰지 않는 가족 구성원도 포함될 수 있으므로, 온디바이스 추론을 기본 동작으로 강제하면 서비스 안정성과 접근성이 오히려 낮아질 수 있다.

따라서 하이브리드 분산 추론의 의미를 "모든 기기에서 로컬 LLM을 직접 실행하는 구조"로 좁게 정의하기보다, 개인화 자산과 추론 책임을 상황에 따라 나누는 구조로 재해석하는 것이 더 적절하다고 보았다. 즉, 분산되는 대상은 반드시 모델 실행 위치만이 아니라, 개인 페르소나 문서, 최근 대화 요약, 로컬 캐시, 권한 필터, 민감도 판단, retrieval 후보군 등도 포함될 수 있다. 이 관점에서 보면 저사양 기기에서는 메인모델 또는 가족 금고 노드의 추론을 기본값으로 사용하고, 개인 페르소나와 대화 맥락은 프롬프트 패키지로 주입하는 방식이 더 안정적이다. 반대로 고사양 기기나 오프라인 상황, 프라이버시가 특히 중요한 질의에서는 온디바이스 Gemma 계열 모델을 선택적으로 사용하는 방식이 적합하다고 판단하였다.

이에 따라 최종 구조는 "하이브리드 분산 추론"이라는 표현보다 "하이브리드 개인화 추론 아키텍처"에 가깝게 정리할 것을 건의한다. 기본 모드는 메인모델과 대화하되, 사용자별 Markdown 페르소나, 최근 대화 요약, 가족방 retrieval evidence를 조합해 개인화된 응답을 생성하는 방식이 적절하다. 선택 모드에서는 기기 성능이 충분한 경우 온디바이스 모델을 사용하고, 네트워크가 불안정하거나 민감한 데이터가 포함된 경우에는 로컬 요약 또는 제한된 retrieval만으로 응답을 보조하는 방안을 제안한다. 이 방식은 보급형 기기에서도 서비스가 느려지지 않도록 하면서, 향후 Gemma 4, Gemma 3n, EXAONE 등 모델이 바뀌더라도 페르소나와 기억 자산을 재사용할 수 있다는 장점이 있다.

최종적으로 이번 연구에서는 디지털 유산 서비스의 장기적 안정성을 위해 "공용 정본 데이터층"과 "개인화된 대화층"을 분리하는 방향을 건의한다. 메인 추론 모델은 가족 금고 노드나 서버 측에 두어 정본 기록 기반의 안정적인 응답을 담당하게 하고, 각 사용자는 기기 성능과 상황에 따라 메인모델 기반 개인화 모드 또는 온디바이스 모델 모드를 선택할 수 있도록 하는 방향이 현실적이라고 판단하였다. 이 구조는 최근 공개되는 Gemma 4 같은 온디바이스 모델의 장점을 활용하면서도, 고령 사용자나 보급형 기기 사용자를 배제하지 않고, 특정 모델 하나에 서비스 전체를 종속시키지 않는다는 점에서 향후 확장성과 유지보수 측면에서도 유리한 대안으로 제안할 수 있다.

추가로 2026년 5월 11일에는 실제 로컬 Ollama 환경에서 Gemma 4 E2B를 가족 기록 더미데이터 기반 질의응답에 연결해 소규모 런타임 실험을 진행하였다. 실험 질문은 송년회 발화 요약, 김치찌개 메모의 단계별 재구성, 기록에 없는 여행 세부정보의 거절 여부로 구성하였다. 그 결과 Gemma 4 E2B는 세 질문 모두에서 비교적 안정적인 답변을 생성하였고, 특히 발화 요약과 레시피 구조화에서는 가족 기록 근거를 무난하게 유지하였다. 다만 세부정보 거절 질문에서는 식당 이름은 없다고 답하면서도 기록 timestamp를 실제 여행 날짜처럼 오해하는 사례가 확인되었는데, 이는 기록 기반 답변이더라도 메타데이터와 사건 본문을 명확히 구분하도록 retrieval 패키지를 더 다듬을 필요가 있음을 보여준다. 또한 응답 속도는 대체로 12초 이상 걸려 체감상 느리게 느껴질 수 있었으며, 따라서 실제 사용자 모드에서는 항상 온디바이스를 기본값으로 두기보다 메인모델 기반 개인화 모드와 선택형 온디바이스 모드를 병행하는 편이 더 현실적이라는 판단을 보강하였다.

같은 날 비교군으로 Qwen 3 0.6B, Qwen 3 4B, EXAONE 3.5 2.4B도 동일한 프레임에서 실험하였다. 그 결과 Qwen 3 0.6B는 일부 질문에서 2~3초대로 가장 빨랐지만, 기록의 timestamp를 실제 여행 날짜로 오해하거나 광안리를 식당 이름처럼 잘못 읽는 등 근거 해석 안정성이 크게 떨어졌다. 반면 Qwen 3 4B는 기록 기반 문장 구성과 없는 정보 거절 면에서는 상당히 안정적이었으나, 응답 시간이 20초에서 70초까지 길어져 현재 데모 기본값으로 쓰기에는 부담이 컸다. EXAONE 3.5 2.4B는 한국어 문장 흐름은 비교적 자연스러웠지만, 송년회 발화나 레시피 정리에서 기록에 없는 해석과 조리 단계를 덧붙이는 경향이 나타났다. 즉 초소형 모델은 속도는 좋지만 안정성이 떨어지고, 중간급 모델은 품질은 좋아지나 시간이 길어지거나 근거 확장 성향이 드러나는 양상이 확인되었다. 이 결과를 종합하면 현재 단계의 가장 현실적인 기본값은 여전히 Gemma 4 E2B이며, 후속 비교 우선순위는 같은 계열의 상위 모델인 Gemma 4 E4B에 두는 편이 타당하다. 향후 추가 비교에서는 단순 정확도보다 “여러 retrieval evidence를 얼마나 자연스럽게 병합하는가”, “한국어 생활 문맥을 얼마나 덜 딱딱하게 표현하는가”, “근거가 없을 때 얼마나 보수적으로 거절하는가”, “프롬프트 패키지를 숨긴 채 최종 답변만 안정적으로 추출하는가”를 중심으로 평가하는 것이 적절하다고 판단하였다. 세부 비교 메모와 실행 스크립트는 별도 문서로 분리하였다.
