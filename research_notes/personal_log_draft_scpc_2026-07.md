## [항목 1]

**연구 내용**: SCPC 2026 AI 에이전트 하네스 대회 참가를 통한 LLM 판단 한계 실측 및 규칙 기반 하네스 설계 방법론 도출

**작성자**: 조현준 **일자**: 2026년 7월 12일 **번호**: (다음 번호)

**참고**:
- 2026 Samsung Collegiate Programming Challenge : AI 챌린지 (DACON 236730), 최종 Public 0.8636
- 개인 저장소 samsung_dacon (규칙 하네스 소스·실험 로그 HANDOFF.md·재현 노트북)
- Geirhos, R. et al. (2020). Shortcut Learning in Deep Neural Networks. Nature Machine Intelligence.
- Ribeiro, M. T. et al. (2020). Beyond Accuracy: Behavioral Testing of NLP Models with CheckList. ACL.
- Gorishniy, Y. et al. (2025). TabM: Advancing Tabular Deep Learning with Parameter-Efficient Ensembling. ICLR.
- GPT-5 API(오프라인 분석용), Ollama 로컬 모델군, DGX Spark 실험 환경

1. 대회 개요 및 참가 목적
   1.1. 가상 개인비서 태스크(700문항)에 대해 focal/target/control 등 5개 판단을 생성하는
        '하네스'를 설계하는 대회로, LLM API 호출이 금지되고 순수 파이썬 결정론 코드만
        허용되어, 사실상 "LLM 없이 LLM급 판단을 재현"하는 문제였다. 본 서비스(로컬 LLM
        기반 유산 관리)의 핵심 질문인 "LLM의 판단을 어디까지 규칙으로 대체할 수 있는가"를
        실측할 기회로 판단하여 개인적으로 참가하였다.

2. LLM의 판단 한계에 대한 실측 결과
   2.1. 지시문은 메커니즘이 아님을 확인하였다. 오프라인 분석에서 GPT-5에 규칙 전문을
        주입하고 "표면에 꽂히지 말고 우선순위대로 판단하라"는 정밀 프롬프트를 적용해도
        정확도가 0.645에서 변하지 않았으며, 오류 유형별 프롬프트 수정은 한 오류를 고치면
        다른 오류가 생기는 풍선 효과만 보였다. 흥미롭게도 출제 측이 문제 프롬프트 안에
        "표면 문장만 보지 말고 판단하라"는 경고문을 이미 포함하고 있었는데, 그 경고문이
        있어도 LLM은 동일하게 실패하였다.
   2.2. LLM은 지각(perception)과 제어흐름(control flow)에서 상반된 성능을 보였다. "무엇이
        적혀 있는가"를 묻는 focal 식별은 충분한 정보를 주면 100% 정확했으나, "어떤 신호가
        어떤 신호를 이기는가"를 묻는 우선순위 판단은 연상적 계산으로 근사하다 붕괴하였다.
        이로부터 "LLM은 지각, 규칙이 제어흐름"이라는 역할 분리 원칙을 도출하였다.
   2.3. 소량 라벨 환경에서 학습 기반 접근의 천장을 실측하였다. 공개 라벨 120개 기준으로
        규칙 하네스는 정확도 1.00, 동일 피처의 GBDT/MLP/TabM(ICLR 2025)은 0.63~0.73에
        머물렀다. 원인은 판단 규칙의 분기 다수가 라벨 1~2건에만 나타나는 희귀 분기라는 점
        으로, 규칙은 1건짜리 분기를 정확히 인코딩하지만 통계 학습은 불가능하였다. 또한
        규칙이 라벨링한 700건으로 학습한 모델은 학습 분포에서 선생을 100% 모사하고도
        평가셋에서 0.64~0.90에 그쳐, 증류로는 선생(규칙)을 넘을 수 없음을 확인하였다
        (자체 파인튜닝 3B 모델도 동일하게 규칙 미달). 데이터의 '양'이 아니라 '정답 라벨의
        출처'가 병목이라는 결론을 얻었다.

3. 도출한 방법론 (본 프로젝트 이식 가능)
   3.1. 지문화×교차대조: 프롬프트에서 가변 슬롯을 제거해 문장을 지문(fingerprint)으로
        만들고 가족(family) 단위로 군집한 뒤, 정답이 있는 셋과 교차대조하여 규칙이 놓치는
        의역 가족을 색출하는 방법. 이 방법으로 평가 점수를 0.705에서 0.8636까지 끌어올렸다
        (절 표현 가족 → 이력 문장 가족 → 지시문 가족의 3연속 발견).
   3.2. 게이트-먼저 규율: 수정 전에 측정 도구(회귀 게이트, 메타모픽 테스트, 결정성 검사,
        무라벨 자가감사)를 먼저 구축하고, 모든 변경을 게이트 통과 후에만 채택하는 개발
        규율. 대회 기간 중 잘못된 개선 시도 다수가 커밋 전에 게이트에서 기각되었다.
   3.3. 소량 앵커 신뢰 원칙: 정답 라벨이 1~2건뿐인 분기라도 그 라벨을 신뢰해야 함을
        실측하였다. 마지막 날 약앵커 분기를 뒤집는 시도 6회가 전부 점수 하락으로 귀결되어,
        생성된 벤치마크에서 앵커는 개수와 무관하게 규칙을 정확히 반영함을 확인하였다.

4. 시행착오 기록
   4.1. 한국어 부정문 오발화: '남기다'를 로컬 처리 단서로 등록했더니 "남기지 않는다"는
        부정문까지 긍정으로 오독하는 결함이 발생, 단서 직후 '-지 않/말/마' 검사를 두어
        해결하였다. 반대로 확인-요구 단서에는 "확인하지 않으면 처리할 수 없다"류 이중부정
        이 많아 같은 가드를 적용하면 오히려 파괴됨을 확인하였다 — 부정 처리도 품사가 아닌
        의미 단위로 판단해야 했다.
   4.2. 하드코딩과 일반화의 경계: 문항별 정답 암기는 비공개 평가에서 실격 사유이자 실제로
        일반화를 해쳤으나, 도메인 어휘 단서 목록(열거)은 오탐 0 검증을 거치면 유효한 일반화
        수단이었다. "열거가 문제가 아니라 검증 없는 열거가 문제"라는 기준을 세웠다.

## [항목 2]

**연구 내용**: SCPC 방법론의 본 서비스 이식 — 규칙 게이트 기반 할루시네이션 차단 구조 설계 및 소형 모델 실측 검증

**작성자**: 조현준 **일자**: 2026년 7월 14일 **번호**: (다음 번호)

**참고**:
- 팀 저장소 ambient-legacy: backend/app/ai/gates/ (구현), research_notes/gate_ab_report_2026-07-12.md (실측 리포트), HALLUCINATION_GATE_PLAN.md (설계)
- DGX Spark + Ollama: gemma4:e2b/e4b, gemma3:27b, exaone3.5:32b, 자체 파인튜닝 qwen2.5-32b-v4
- 골든셋 30문항 자체 제작 (answerable/unanswerable/conflict/advice 라벨, 규칙 채점기)

1. 문제 재정의
   1.1. 기존에는 응답 할루시네이션을 모델 성능 향상(모델 대형화, 32B 파인튜닝)으로 해결
        하려 하였으나, 대회에서 실측한 "지시문은 메커니즘이 아니다"라는 결론에 따라 접근을
        전환, 모델은 그대로 두고 LLM이 결정하는 영역을 규칙으로 줄이는 구조를 설계하였다.
   1.2. 진단 결과 기존 파이프라인의 할루 유입점은 세 곳이었다: 검색 결과가 빈약해도 무조건
        LLM을 호출하는 구조, 프롬프트 지시문에만 의존하는 방어, 그리고 응답 무검증 통과.

2. 게이트 구조 설계 및 구현
   2.1. 입력 래더: 질문이 기록으로 답변 가능한지 LLM 호출 전에 규칙으로 판정하여
        REFUSE(기록 없음 거절)/CLARIFY(재질문)/CONFLICT(기록 모순 양립 제시)/ANSWER로
        라우팅하였다. ANSWER 외 경로는 LLM을 호출하지 않으므로 해당 경로의 할루시네이션은
        구조적으로 0이다. 이는 대회 하네스의 판단 래더(proceed/amend/hold/ask)와 동형이다.
   2.2. 출력 게이트: LLM 응답을 문장 단위로 분해하여 검증 가능한 원자(숫자·날짜·시각·
        인용구)가 검색된 기록에 실재하는지 대조하고, 미지원 원자를 포함한 문장을 차단,
        전량 차단 시 기록 인용 응답으로 강등하였다. 내부 추론 태그·프롬프트 라벨 누수와
        비한국어 응답도 규칙으로 제거하였다.
   2.3. 가족방 엔티티 사전: 대회에서 닫지 못했던 고유명사(이름·장소) 환각을, 본 서비스가
        닫힌 세계(가족 기록 전체와 멤버 정보가 DB에 실재)라는 점을 이용해 해결하였다.
        FamilyMember·User 테이블의 이름과 관계 호칭, 기록 텍스트의 장소 접미사(X식당,
        X병원 등) 추출로 방별 사전을 구축하고, "식당 이름" 류 질문은 해당 카테고리의
        이름 있는 장소가 실재할 때만 답변 가능으로 판정하였다. 특정 가족의 데이터는 코드에
        넣지 않는다 — 규칙은 코드에, 사전은 데이터에 두어 기록 추가 시 재학습 없이 즉시
        반영된다.

3. 실측 결과 (골든셋 30문항, 동일 모델 게이트 전/후 비교)
   3.1. 최소형 gemma4:e2b(2B급) 기준 전체 정확도 0.63 → 0.97, 거절 정확도 0.31 → 1.00,
        할루율·누수율·빈응답률 모두 0으로 수렴하였다. 5개 모델 전체에서 게이트 적용 시
        평균 +23%p 이상 향상되었고 할루율은 전 모델 0~3%로 소멸하였다.
   3.2. 핵심 결과로, 2B급 온디바이스 모델에 게이트를 결합한 성능(0.97)이 자체 파인튜닝한
        32B 모델 단독 성능(0.90)을 상회하였다. 파인튜닝(수 주간의 학습 비용) 노선이 사주던
        개선을 학습 비용 0의 결정론 규칙이 대체·상회함을 확인하였으며, 이는 본 서비스의
        온디바이스·데이터 주권 방향(Sovereign AI)과 정확히 부합한다. 파인튜닝의 역할은
        사실 암기가 아니라 말투·거절 형식·페르소나 정합으로 재배치하는 것이 타당하다.

4. 시행착오 기록
   4.1. 소형 Gemma 계열이 raw 프롬프트 엔드포인트에서 전량 빈 응답을 반환하는 문제를
        발견, 챗 템플릿을 적용하는 chat 엔드포인트로 전환하여 해결하였다. 이 과정에서 빈
        응답이 채점기에서 '정답' 처리되는 채점 맹점도 발견·수정하였다 — 평가 도구 자체도
        테스트 대상이라는 교훈을 얻었다.
   4.2. "어느 가게" 류 질문의 '어느'가 단서 목록에 누락되어 래더가 사실질문으로 인식하지
        못하는 결함을 자가테스트가 검출하였다. 게이트 층은 순수 표준 라이브러리 함수로
        작성하여 DB·프레임워크 없이 단독 회귀 테스트(16케이스)가 가능하도록 유지하였다.

## [항목 3]

**연구 내용**: 대회 품질 인프라(회귀 게이트·메타모픽 검증·재현성 규율)의 본 서비스 이식 및 한국어 질의 정규화 적용

**작성자**: 조현준 **일자**: 2026년 7월 14일 **번호**: (다음 번호)

**참고**:
- Ribeiro, M. T. et al. (2020). Beyond Accuracy: Behavioral Testing of NLP Models with CheckList. ACL. (메타모픽 검증의 이론적 배경)
- 팀 저장소 ambient-legacy: backend/scripts/selftest_gates.py (22케이스), backend/app/ai/gates/textrules.py
- 대회 저장소 samsung_dacon: tools/check.py(CI 게이트), tests/test_metamorphic.py, tools/self_audit.py

1. 대회에서 LLM 판단 규칙 외에 별도로 구축했던 품질 인프라를 본 서비스에 이식하였다.
   1.1. 메타모픽 검증: "의미를 바꾸지 않는 입력 변형에 출력이 불변해야 한다"는 성질 기반
        테스트를 게이트 층에 적용하였다. 검색 결과 순서 셔플 불변, 무관 기록 주입 시 거절
        판정 유지, 동일 입력 2회 실행 시 완전 동일 출력(결정성), 빈 문자열 chunk 생존성의
        6개 성질을 자가테스트로 고정하였다. 대회에서 이 방식은 라벨 없이도 회귀를 검출하는
        핵심 수단이었으며(순서 셔플 700문항 0건 변동 검증 등), 본 서비스에서도 골든셋 라벨
        없이 게이트 결함을 조기 검출하는 안전망이 된다.
   1.2. 결손 입력 방어: 대회 제출 직전 심층 점검에서 필드 하나(turn_index)가 null이면 전체
        실행이 중단되는 결함을 발견해 수정했던 경험을 반영, 게이트 층이 빈 문자열·결손
        chunk에도 중단 없이 동작함을 테스트로 보장하였다. 단일 입력 결함이 서비스 전체
        실패로 번지지 않게 하는 방어적 파싱 원칙이다.
   1.3. 한국어 질의 정규화 이식: 대회에서 검증한 표기 접기(메신저 종결 변이 '알려주세여/
        알려주세요→알려줘', 맞춤법 혼동 '됬→됐' 등 단방향 안전 목록)와 공백 무시 단서
        매칭('몇 시'/'몇시' 통합)을 질의 전처리에 적용하였다. 가족 구성원의 구어체·오탈자
        질의에 대한 판정 강건성을 확보하면서 단서 목록의 중복 등재를 제거하였다.
   1.4. 측정 규율 이식: 변경 전 반드시 측정 도구를 먼저 만들고(골든셋 30문항 + 규칙 채점기),
        모든 변경을 전/후 비교와 변경 코호트 검사(flip census — 변경이 의도한 문항에서만
        발생했는지)로 채택하는 개발 규율을 유지하였다. 대회에서 이 규율은 그럴듯하지만
        잘못된 개선 다수를 커밋 전에 기각시켰다.

2. 이식 결과 게이트 자가테스트는 22케이스(래더 4, 출력 게이트 4, 엔티티 사전 8, 메타모픽
   6)로 확장되었으며 전부 통과 상태다. 순수 표준 라이브러리 함수로만 구성되어 DB·프레임
   워크 없이 1초 내 실행 가능해, 이후 팀원 누구든 게이트 수정 시 즉시 회귀 확인이 가능하다.

## [주간연구일지 발췌용] — 해당 주차 칸에 붙여넣기

**전주 목표 및 진행사항 : (실적/목표 - %)**

목표: RAG 응답 할루시네이션 저감 (기존: 모델 대형화·파인튜닝 접근)
진행사항: (실적/목표 - 100%)
ㆍSCPC 대회에서 검증한 '규칙 게이트' 방식으로 접근 전환 — 입력 래더(답변 불가 질문은
LLM 미호출 거절)·출력 게이트(응답의 숫자/날짜/인용/고유명사를 기록과 대조, 미지원 문장
차단)·가족방 엔티티 사전(멤버 테이블+기록에서 자동 구축) 구현 및 백엔드 통합 완료.
ㆍ자체 골든셋 30문항(모델 5종 × 게이트 전/후) 실측: 전체 정확도 평균 +23%p, 할루시네이션
율 전 모델 0~3%로 소멸. 특히 2B급 온디바이스 모델(Gemma E2B)+게이트가 0.97로, 자체
파인튜닝 32B 모델 단독(0.90)을 상회 — 온디바이스·데이터 주권 방향의 실측 근거 확보.
ㆍ게이트 자가테스트 22케이스(메타모픽 검증 포함) 구축, 팀 저장소 반영 완료.

**금주 계획 및 논의사항 :**

1. 게이트 구조의 실서비스 검증
ㆍ실제 업로드 데이터 기반 골든셋 확장(현 30문항 → OCR/STT 실데이터 포함) 및 재측정.
ㆍ과잉거절 경계 케이스(소형 모델 빈 응답 재시도 후에도 실패하는 문항) 분석.
2. 파인튜닝 역할 재정의 논의
ㆍ사실 암기용 파인튜닝은 기록 추가 시 낡는 문제 확인 → 말투·거절 형식·페르소나 정합
전용 경량 파인튜닝(게이트 통과 응답을 학습 데이터로 활용)으로 전환 검토.

**특기 사항 및 문제점 :**

문제점: 고유명사 환각 중 '이름 있는 장소'는 접미사 규칙으로 차단되나, 접미사 없는 고유명사
(상호명 단독 등)는 여전히 검출 한계. 엔티티 사전의 커버리지가 업로드 태깅 품질에 의존.
해결 방향: OCR/STT 파이프라인에 엔티티 후보 추출 단계를 추가해 업로드 시점에 사전을
풍부화하는 방안 검토 중. 파인튜닝 대비 게이트의 비용 우위(학습 0원, 기록 추가 즉시 반영)가
실측으로 확인되어, 팀 리소스를 모델 학습에서 데이터 파이프라인 품질로 재배치 제안 예정.
