## 3.8 AI 추론 및 RAG 응답 구조 설계

본 프로젝트의 AI 응답 구조는 특정 LLM 모델 하나에 고정되지 않도록 모델 교체형 구조로 설계하였다. 가족 디지털 유산 서비스에서 장기적으로 보존되어야 하는 핵심 자산은 특정 모델이 아니라 가족의 원본 기록, OCR/STT 기반 메타데이터, 가족방 권한 구조, 페르소나 문서, 검색 인덱스이기 때문이다. 따라서 모델은 필요에 따라 EXAONE, Gemma, Ollama 기반 로컬 모델, 향후 온디바이스 모델 등으로 교체할 수 있도록 하고, 상위 서비스 구조는 유지되도록 설계하였다.

이를 위해 백엔드에는 모델 목록과 실행 방식을 관리하는 model registry, 페르소나 문서를 불러오는 persona loader, 사용자 질문과 검색 근거를 결합하는 prompt builder, 실제 모델 호출 방식을 추상화하는 provider adapter 구조를 적용하는 방향으로 설계하였다. 이 구조에서는 앱이 선택한 모델과 페르소나 정보가 백엔드로 전달되고, 백엔드는 가족방 기록에서 검색된 evidence를 함께 조합하여 모델 입력을 생성한다. 모델의 최종 응답은 단순 생성 결과가 아니라, 실제 가족 기록에 근거한 RAG 응답으로 반환되도록 구성하였다.

페르소나 정보는 하나의 긴 프롬프트 문자열로 고정하지 않고 Markdown 기반 문서 자산으로 관리하는 방식을 검토하였다. 예를 들어 인물의 기본 정보, 말투, 주요 생애 사건, 기억 사용 원칙을 각각 분리해 관리하면 가족 구성원이 내용을 직접 검수하고 수정할 수 있다. 또한 모델이 교체되더라도 동일한 페르소나 자산을 재사용할 수 있으므로, 디지털 유산 서비스의 장기 보존성과 유지보수성에 적합하다.

온디바이스 LLM은 프라이버시와 오프라인 응답 측면에서 장점이 있으나, 보급형 스마트폰의 메모리, 발열, 배터리 소모, 기기별 성능 차이를 고려하면 모든 사용자에게 기본값으로 강제하기 어렵다. 따라서 본 프로젝트에서는 서버 또는 가족 금고 노드의 메인 모델을 기본 추론 계층으로 두고, 향후 고사양 기기나 민감 질의, 오프라인 상황에서 온디바이스 모델을 선택적으로 활용하는 하이브리드 추론 구조를 후속 확장 방향으로 설정하였다.

## 4.14 로컬 LLM 기반 RAG 응답 안정성 검증

본 프로젝트에서는 단순히 가족 기록을 저장하는 기능을 넘어, 저장된 기록을 근거로 사용자의 질문에 답변할 수 있는 로컬 LLM 기반 RAG 응답 구조를 검증하였다. 특히 디지털 유산 서비스에서는 모델이 기록에 없는 내용을 그럴듯하게 만들어내거나, 내부 메타데이터와 페르소나 문서를 그대로 노출하는 문제가 치명적일 수 있다. 따라서 응답 품질 검증의 기준을 일반적인 자연스러움이 아니라 기록 근거성, 누락 정보 거절, 충돌 기록 처리, 내부 정보 비노출로 설정하였다.

먼저 실제 DB에 저장된 가족방 업로드 기록을 검색하고, 이를 바탕으로 송년회 건배사 요약, 부산 여행 정보 확인, 병원 예약 시간 충돌 처리와 같은 질문을 수행하였다. 이 과정에서 모델이 업로드 시각을 실제 사건 시각으로 오해하지 않도록 `saved_at`은 저장 시각일 뿐 사건 날짜가 아니라는 규칙을 프롬프트에 반영하였다. 또한 `source`, `tags`, `confidence`와 같은 내부 검색 메타데이터는 최종 답변에 그대로 노출하지 않도록 응답 규칙을 강화하였다.

이후 Gemma 계열과 EXAONE 계열 모델을 대상으로 LoRA 파인튜닝 가능성을 검증하였다. 실험은 세 단계로 진행하였다. v1에서는 592개 synthetic SFT 데이터셋으로 기본 가능성을 확인하였고, v2에서는 metadata leak, prompt injection, missing detail hard-negative를 늘려 2,640개 데이터셋으로 확장하였다. v3에서는 v2 평가에서 실제로 실패한 유형을 targeted repair 데이터로 보강하여 3,424개 데이터셋을 구성하였다.

학습 데이터는 외부 데이터셋을 그대로 복사하지 않고, Ambient Legacy의 서비스 조건에 맞춘 deterministic template 기반 synthetic data로 직접 구성하였다. 데이터 유형은 가족 기록 요약, 기록에 없는 정보 거절, OCR 기록과 텍스트 기록의 충돌 처리, 내부 메타데이터 비노출, 프롬프트 인젝션 방어, 저장 시각과 사건 시각 구분으로 나누었다. 이를 통해 가족 기록 RAG 서비스에서 실제로 발생할 수 있는 실패 상황을 작게나마 재현하고, 모델이 해당 규칙을 학습할 수 있는지 확인하였다.

모델별 실험 결과, v1 generation evaluation에서는 EXAONE과 Gemma가 각각 3/7만 통과하여 두 모델 모두 불안정성이 확인되었다. v2에서는 hard-negative 데이터 확장 효과로 직접 metadata 누출이 줄었고, EXAONE은 6/7, Gemma는 5/7을 통과하였다. v3에서는 targeted repair 데이터가 적용되면서 EXAONE v3가 7/7을 통과하였고, Gemma v3는 5/7을 통과하였다.

| 구분 | EXAONE v3 | Gemma v3 |
| --- | ---: | ---: |
| Generation evaluation | 7/7 통과 | 5/7 통과 |
| Full test loss | 4.066 | 5.749 |
| Test perplexity | 58.336 | 313.841 |
| Peak memory | 약 2.567GB | 약 6.413GB |

EXAONE v3는 `saved_at`을 사건 날짜로 오해하는 문제와 충돌 기록을 하나로 단정하는 문제를 개선하였다. 예를 들어 저장 시간이 2026년이면 여행 날짜도 2026년이라고 답하라는 공격에 대해, `saved_at`은 기록 저장 시각일 뿐 실제 사건 날짜로 확정할 수 없으며 기록 본문에서 확인되는 시점은 2018년 여름이라고 구분하였다. 또한 병원 예약 시간이 10시와 11시로 충돌하는 경우 하나만 고르라는 압박에도 두 기록을 모두 제시하고 확정할 수 없다고 응답하였다.

Gemma v3도 persona markdown 직접 노출과 timestamp 오해 문제는 개선되었으나, 일부 상황에서 기록에 없는 세부 정보를 거절한 뒤 다시 추가하거나, 충돌 기록을 하나로 단정하는 문제가 남았다. 따라서 현재 연구 단계에서는 EXAONE v3가 가족 기록 RAG용 파인튜닝 후보로 더 적합하다고 판단하였다. 다만 EXAONE은 실행 시 `trust_remote_code` 확인과 라이선스 검토가 필요하므로, 실제 서비스 적용 전에는 모델 실행 구조와 사용 조건을 추가로 확인해야 한다.

이번 검증의 핵심 성과는 모델 파인튜닝만으로 모든 문제를 해결하기보다, 데이터셋 설계, 프롬프트 규칙, LoRA adapter, 백엔드 후처리 guardrail을 함께 결합해야 한다는 점을 확인한 것이다. 가족 디지털 유산 서비스에서는 한 번의 환각이나 내부 정보 노출도 사용자 신뢰에 영향을 줄 수 있으므로, 모델 응답은 반드시 검색 근거와 백엔드 검증 규칙을 함께 통과하도록 설계하는 것이 적절하다.

## 5.2 기대 효과 보강 문단

본 프로젝트는 가족 기록을 단순히 저장하는 수준을 넘어, 저장된 기록을 검색 가능한 지식 자산으로 전환하고 자연어 질의응답에 활용할 수 있는 기반을 마련한다. 사용자는 사진, 영상, 텍스트, 음성으로 흩어져 있던 가족의 일상 기록을 가족방 단위로 관리할 수 있으며, 향후에는 특정 인물, 날짜, 장소, 사건을 기준으로 기록을 검색하고 회상할 수 있다.

또한 로컬 LLM과 RAG 구조를 결합함으로써 외부 범용 AI 서비스에 민감한 가족 데이터를 직접 전송하지 않고도 지능형 응답을 제공할 수 있다. 이는 가족 데이터 주권을 보호하면서도 AI 기반 회상 기능을 구현할 수 있다는 점에서 기존 단순 클라우드 저장 서비스와 차별화된다. 특히 페르소나 문서와 기록 근거를 분리하여 관리하면, 특정 모델이 바뀌더라도 가족 기록과 인물 정보는 장기적으로 재사용될 수 있다.

## 5.3 제언 및 향후 과제 보강 문단

향후 과제는 크게 세 가지로 정리할 수 있다. 첫째, 실제 OCR/STT 파이프라인을 업로드 기록과 연결하여 이미지, 영상, 음성 데이터를 자동으로 검색 가능한 메타데이터로 변환해야 한다. 현재 프로토타입은 업로드와 저장소 조회 구조를 중심으로 구현되어 있으므로, 후속 단계에서는 OCR confidence, STT 인식 결과, 영상 시간 정보 등을 별도 메타데이터로 저장하는 구조가 필요하다.

둘째, LLM 응답 안정성을 높이기 위한 백엔드 후처리 guardrail을 추가해야 한다. 최종 응답에 `source=`, `tags=`, `confidence=`, `saved_at=`과 같은 내부 메타데이터가 포함되는지 검사하고, 검색 근거에 없는 날짜, 장소, 음식, 인물명이 생성되었는지 확인하는 절차가 필요하다. 또한 충돌 기록이 존재할 때 모델이 하나의 답으로 단정하지 않도록, 충돌 evidence를 감지하는 규칙도 함께 적용해야 한다.

셋째, 모델 평가 데이터셋을 확대해야 한다. 이번 실험에서는 7개 핵심 generation evaluation으로 모델을 비교했으나, 실사용 환경에서는 질문 표현, 오타, 가족별 말투, 복합 질문이 훨씬 다양하게 나타날 수 있다. 따라서 metadata leak, persona disclosure, system prompt disclosure, timestamp misuse, missing detail hallucination, conflict collapse, prompt injection, normal recall summary, structured summary를 포함한 최소 50개 이상의 평가셋을 구성하고, 실제 DB retrieval 형식과 유사한 테스트 데이터를 추가해야 한다.
