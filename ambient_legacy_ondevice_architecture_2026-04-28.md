# Ambient Legacy 구조 재설계 제안

## 결론

내 판단으로는 `완전 서버형`도, `완전 단말 단독형`도 최적해가 아니다.  
가장 적절한 방향은 아래의 `하이브리드 분산형 Sovereign AI` 구조다.

- 평상시 응답: 각 사용자 기기에서 온디바이스 모델로 처리
- 가족 유산 원본과 정본 인덱스: 가족 금고 노드 또는 로컬 홈 서버가 관리
- 장기 보존: 암호화된 클라우드 백업

즉, 서비스 철학은 `내 데이터는 내 쪽에서 유지`를 지키되,  
실제 운영은 `한 기기 망가짐 = 유산 소실`이 되지 않도록 분산한다.

## 왜 이 방향이 더 맞는가

### 1. 유산 서비스의 핵심 리스크와 더 잘 맞음

디지털 유산 서비스는 일반 챗앱과 달리 `장기 보존성`이 핵심이다.

- 스마트폰 교체
- 배터리 노후화
- OS 업데이트 중단
- 앱 삭제
- 저장소 손상

이런 일이 필연적으로 생긴다.  
그래서 "온디바이스만" 믿는 구조는 철학적으로는 좋아도, 유산 서비스로는 불안하다.

반대로 전부 서버형으로 가면:

- 데이터 주권 메시지가 약해짐
- 가족 기록이 중앙 인프라에 과도하게 의존
- 민감 데이터가 외부 스택에 묶임

그래서 정답은:

- `추론은 단말 가까이`
- `정본은 가족 쪽`
- `백업은 암호화 후 외부`

이 3단 구조다.

## 권장 아키텍처

## 1. 계층 구조

### A. Personal Edge Node

각 가족 구성원의 폰, 태블릿, 노트북.

역할:

- 온디바이스 모델 실행
- 개인 캐시 인덱스 보관
- 최근 열람/최근 대화 기반 빠른 RAG
- 페르소나 Markdown 로드
- 이미지/오디오/텍스트 로컬 전처리

권장 모델:

- `Gemma 3n E2B/E4B`
- 경량 텍스트 전용 보조 모델

### B. Family Vault Node

가족 단위의 정본 노드.  
형태는 미니 PC, 맥미니, NAS 옆의 서비스 노드, 로컬 서버 등이 가능하다.

역할:

- 원본 유산 데이터 정본 저장
- OCR/STT/메타데이터 파이프라인
- 가족 공용 인덱스 생성
- 권한 필터링
- 페르소나 버전 관리
- 단말 간 동기화

권장 모델:

- `EXAONE` 또는 더 큰 Gemma 계열
- 고정밀 재질의 응답이 필요할 때만 서버 측 사용

### C. Encrypted Cold Backup

Google Drive, Cloud Storage, S3 유사 저장소 등.

역할:

- 암호화된 blob 백업
- 모델/인덱스 스냅샷 백업
- 단말 복구 시 재시드

원칙:

- 복호화 키는 외부 저장소에 두지 않음
- 외부는 `금고`이지 `추론 주체`가 아님

## 2. 추론 경로

### 경로 A. 빠른 개인 응답

```text
질문
-> 개인 기기
-> 로컬 인덱스 검색
-> persona.md 로드
-> 온디바이스 Gemma 응답
```

이 경로는:

- 오프라인 가능
- 빠름
- 사적 대화에 적합

### 경로 B. 가족 정본 기반 응답

```text
질문
-> 개인 기기
-> 권한 확인
-> Family Vault Node 검색
-> 관련 청크/메타데이터 반환
-> 기기 또는 금고 노드에서 최종 생성
```

이 경로는:

- 더 정확한 검색 가능
- 최신 정본 반영
- 장기 기록 회상에 유리

### 경로 C. 복구/기기 교체

```text
새 기기 로그인
-> 암호화 백업 복원
-> persona.md / 인덱스 / 메타데이터 재동기화
-> 온디바이스 모델 재배치
```

이 경로가 있어야 "기기 노후" 리스크가 보완된다.

## Gemma를 고려할 가치가 큰 이유

공식 Google 문서 기준으로 현재 확인되는 점은 다음과 같다.

- `Gemma 3n`은 폰, 태블릿, 노트북 같은 일상 기기용으로 최적화된 모델이다.
- `Gemma 3n`은 텍스트뿐 아니라 이미지와 오디오 입력도 다룰 수 있다.
- Android용 `LLM Inference API`는 이미지/오디오를 텍스트와 함께 세션에 넣는 멀티모달 입력을 지원한다.
- Android용 `AI Edge RAG SDK`도 제공되지만, 공식 문서상 `under active development` 상태다.

이 말은 곧:

- `온디바이스 멀티모달 추론`은 충분히 시도할 가치가 있음
- `온디바이스 RAG`도 프로토타입 가능
- 다만 정식 서비스 핵심을 전부 여기에만 걸기에는 아직 이르다

그래서 Gemma는 `1차 사용자 단말 추론 모델`로 매우 유망하지만,  
정본 관리와 장기 보존까지 단말 하나에 몰아주는 건 피하는 게 좋다.

## EXAONE vs Gemma 판단

### Gemma 쪽이 강한 부분

- 온디바이스 지향성이 더 명확함
- 멀티모달 입력 구조가 공식적으로 정리돼 있음
- 모델 선택형 모바일 UX 레퍼런스가 존재함
- Android 쪽 실험 자산이 풍부함

### EXAONE 쪽이 강한 부분

- 한국어 중심 품질 기대치
- 로컬 서버 추론에서 안정적인 텍스트 응답용 메인 모델로 쓰기 좋음
- 팀이 이미 고려 중인 방향과 연결이 쉬움

### 최종 판단

둘 중 하나만 고집할 필요는 없다.

가장 현실적인 조합은:

- `단말`: Gemma 3n 계열
- `가족 금고 노드`: EXAONE 또는 더 큰 텍스트 모델

이 구조다.

즉, Gemma를 EXAONE 대체재로만 보지 말고  
`단말 추론층`을 맡는 모델로 보면 훨씬 설계가 깔끔해진다.

## 페르소나를 Markdown으로 유지하는 방식은 좋은가

좋다. 다만 `단일 md 한 장`만으로는 부족하고 계층화가 필요하다.

권장 구조:

- `identity.md`: 인물의 고정 정체성
- `voice.md`: 말투, 금기, 화법
- `timeline.md`: 주요 생애 사건
- `relationships.md`: 가족 관계와 호칭
- `memory_policy.md`: 답변 거리두기, 추정 금지 규칙

그 이유는 다음과 같다.

- 장기 수정이 쉬움
- 버전 관리가 쉬움
- 사용자 검수 가능
- 모델 교체 시에도 재사용 가능

즉, `모델은 바뀌어도 페르소나 문서 자산은 남는 구조`가 된다.

이건 유산 서비스에 특히 중요하다.

## 추천 데이터 구조

### 1. Canonical Memory Pack

가족 기록의 표준 패키지.

구성:

- 원본 파일
- OCR/STT/설명 메타데이터
- 권한 정보
- 태그
- 시간 정보
- 인물 연결

### 2. Persona Pack

한 인물 단위의 페르소나 패키지.

구성:

- markdown 파일들
- 검수 상태
- 출처 메모
- 버전

### 3. Device Runtime Pack

각 단말에서 실행되는 런타임 패키지.

구성:

- 선택된 모델 프로필
- persona 조합본
- 최근 인덱스 캐시
- 시스템 프롬프트
- 안전 규칙

## 지금 구조를 어떻게 갈아엎는 게 좋은가

## 1단계

현재의 `앱 -> FastAPI -> 저장소` 구조는 버리지 말고,  
역할을 바꿔야 한다.

기존:

- 서버가 거의 모든 지능 처리 담당

변경:

- 서버는 정본 관리와 무거운 분석 담당
- 기기는 개인 추론과 빠른 응답 담당

## 2단계

앱에 아래 개념을 추가한다.

- 모델 선택 화면
- persona 선택 또는 연결
- 오프라인 가능 여부 표시
- 로컬 인덱스 동기화 상태
- 이 답변이 어떤 레이어에서 생성됐는지 표시

예:

- `이 기기에서 생성됨`
- `가족 금고 기록 기반`
- `백업에서 복원된 기록 포함`

## 3단계

RAG를 `중앙 하나`가 아니라 `다층 RAG`로 본다.

- Layer 1: device cache retrieval
- Layer 2: family vault retrieval
- Layer 3: optional cold archive recall

이렇게 해야 속도와 보존성을 동시에 챙길 수 있다.

## 권장 MVP

지금 바로 구현을 시작한다면 MVP는 이렇게 잡는 게 맞다.

### MVP-A

- 모바일 앱에서 모델 선택 가능
- `Gemma 3n` 온디바이스 채팅 가능
- persona markdown 로딩 가능
- 일부 최근 메모리에 대해 로컬 RAG 가능

### MVP-B

- Family Vault Node에서 OCR/STT/메타데이터 생성
- 공용 검색 인덱스 생성
- 가족 권한 필터 적용

### MVP-C

- 기기 교체 시 persona와 인덱스 일부를 재동기화
- 암호화 백업 복원

이 3개가 모이면 서비스 스토리가 매우 강해진다.

## 만들면서 주의할 점

### 하지 말아야 할 것

- 모든 추론을 온디바이스만으로 해결하려는 설계
- 페르소나를 하나의 거대 프롬프트 문자열로만 관리
- OCR/STT 결과를 원본과 분리하지 않는 설계
- 권한 필터링보다 생성 응답을 먼저 하는 구조

### 꼭 지켜야 할 것

- 정본과 캐시 분리
- 페르소나와 검색 근거 분리
- 기기별 모델 프로필 분리
- 복구 경로를 처음부터 설계

## 내가 추천하는 최종 방향 한 줄

`Ambient Legacy`는 `EXAONE 중심 서버형 앱`으로 가기보다,  
`Gemma 온디바이스 개인 노드 + EXAONE/대형 모델 가족 금고 노드 + 암호화 백업` 구조로 재설계하는 쪽이 더 서비스 성격에 맞다.
